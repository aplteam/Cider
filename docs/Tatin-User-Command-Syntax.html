Tatin User Command Syntax

BuildPackage

Syntax

Create a new version of a package (resulting in a ZIP) from the directory.

Syntax: up to 2 arguments

Accepts modifiers -dependencies=, -version=

]Tatin.BuildPackage [<package-folder>] [<target-folder>] -dependencies= -version=

Details

]Tatin.BuildPackage [<package-folder>] [<target-folder>] -dependencies= -version= Creates a ZIP file from the directory ‚çµ[1] that is a package, and saves it in ‚çµ[2].

Requires directory ‚çµ[1] to host a file ‚Äúapl-package.json‚Äù defining the package.

Always bumps the build number except when -version= is specified and carries a build number.

  • If ‚çµ[2] is not specified Cider (if available) will be questioned: ‚ÄúdistributionFolder‚Äù
  • If ‚çµ[1] is not specified it will act on any open Cider project If BuildPackage makes any assumptions at all then the user will be asked for confirmation.

-version= Use this to set the version number in both the package project and the package that is about to be created. You have several options: * -version=+0.0.1 ‚Üí bumps the patch number * -version=+0.1.0 ‚Üí bumps the minor number, and resets the patch number * -version=+1.0.0 ‚Üí bumps the major number, and resets patch & minor number * -version=1.2.3-beta-2 assigns the given string to ‚Äúversion‚Äù It will preserve any build number and bump it * -version=1.2.3-beta-2+123 will replace whatever is saved on ‚Äúversion‚Äù, including the build number

-dependencies= Use this to specify a subfolder of the project holding package dependencies.

Usually there is no need to specify this, refer to the documentation for details: the document ‚ÄúPublishing Packages‚Äù.

Cache

Syntax

List the contents of the Tatin package cache.

Syntax: up to 1 argument

Accepts modifiers -clear, -force, -path

]Tatin.Cache [url] -path -clear -force

Details

]Tatin.Cache [url] -path -clear -force Lists all packages of all URLs saved in the Tatin package cache.

One may specify a URL as argument; then only packages of that URL are listed.

-path By default just domain names and package names are returned. By specifying this option you can force it to return full paths instead.

-clear Lists all cached packages and then asks for confirmation before removing them.

If confirmed all packages are removed or, if an argument (URL) was specified, all packages of that URL.

-force Ignored in case -clear was not specified as well. Prevents the user from being interrogated regarding clearing the cache. Mainly for test cases.

CheckForLaterVersion

Syntax

Check whether for the installed packages a later versions are available.

Syntax: 1 argument

Accepts modifiers -dependencies, -major

]Tatin.CheckForLaterVersion <install-folder> -major -dependencies

Details

]Tatin.CheckForLaterVersion <install-folder> -major -dependencies Checks whether later versions of the installed packages are available.

Takes a folder that hosts a file ‚Äúapl-buildlist.json‚Äù as argument.

Scans all known Registries with a priority greater than 0 for later versions.

Returns a matrix with five columns for all packages found: [;1] Currently installed package ID [;2] package ID of the latest version found or something like ‚Äúno response‚Äù or ‚Äúnot found‚Äù [;3] URL of the Registry the package was originally installed from [;4] ! for a later version [;5] URL of the Registry hosting a different (usually later) version; might be empty

-major By default later MAJOR versions are ignored, but this default behaviour can be changed by specifying -major: then only later major versions are reported.

-dependencies By default only principal packages are checked.

You may include dependencies by specifying this flag.

DeprecatePackage

Syntax

Declare packages as deprecated.

Syntax: 2 arguments

Accepts modifier -force

]Tatin.DeprecatePackage <URL|[Alias><group><name>[<major-version>] <comment> -force

Details
]Tatin.DeprecatePackage <URL [Alias><group><name>[<major-version>] <comment> -force

Declares a particular major version of a package or all major versions of a particular package as deprecated. Requires two arguments:

  1. Argument
    • If a package ID like ‚ÄúGroupName-PkgName-6 is specified as first argument then a new version is published for the major version number 6 that will have a property ‚Äùdeprecated" injected into its config file with the value 1.

      It will also have a property ‚Äúcomment‚Äù that can carry stuff like ‚ÄúUse xyz instead‚Äù.

    • If a package ID like "GroupName-PkgName is specified then new versions are published for all major versions of that package.
  2. Argument A text that will be injected into the package's config file together with ‚Äúdeprecated‚Äù.

-force By default the user is questioned whether she is sure. With this flag this can be overwritten. Mainly for tests.

FindDependencies

Syntax

Attempts to find all folder(s) with the given Tatin package(s)

Syntax: 1-2 arguments

Accepts modifier -verbose

]Tatin.FindDependencies <comma-separated list of pkgs> [target] -verbose

Details

]Tatin.FindDependencies <comma-separated list of pkgs> [target] -verbose Scans ‚Äútarget‚Äù recursively for a file ‚Äúapl-dependencies.txt‚Äù.

‚Äútarget‚Äù can be a folder or a URL or registry alias or [*] for all defined Registries.

Folders with such a file will be scanned for packages defined in ‚ÄúpkgList‚Äù.

Useful for finding out where one or more packages are used.

  • If ‚Äútarget‚Äù is embraced by square brackets it is interpreted as a Registry alias
  • If it is [*] all defined Registries with a priority greater than 0 are scanned
  • If it starts with http[s]:// it is interpreted as a URL pointing to a Registry
  • Otherwise ‚Äútarget‚Äù is treated as path
  • If ‚Äútarget‚Äù is not specified it becomes the current dir, but confirmation is required

‚ÄúpkgList‚Äù must be a simple char vector with ‚Äú,‚Äù-separated package IDs. The packages can be specified fully or partly. ‚ÄúGroup‚Äù and ‚Äúversion‚Äù can be left out while ‚ÄúName‚Äù is mandatory. You may specify a major version but ‚Äúminor‚Äù and ‚Äúpatch‚Äù will be ignored.

Note that the sequence of the arguments does not matter.

By default the folder(s)/Registries hosting the package(s) are returned, but that can be changed by specifying the -verbose flag:

-verbose By setting this flag you can force the user command to report the actual package folder(s) rather than the hosting folder.

As a side effect the result might be larger, in case a package is installed more than once.

GetDeletePolicy

Syntax

Request which ‚ÄúDelete‚Äù policy is operated by a Registry.

Syntax: up to 1 argument

Accepts modifier -check

]Tatin.GetDeletePolicy [<Registry-URL>] -check

Details

]Tatin.GetDeletePolicy [<Registry-URL>] -check Requests which ‚ÄúDelete‚Äù policy is operated by a Registry.

Return one of ‚ÄúNone‚Äù, ‚ÄúAny‚Äù, ‚ÄúJustBetas‚Äù:

  • ‚ÄúNone‚Äù means a package, once published, cannot be deleted
  • ‚ÄúAny‚Äù means any package can be deleted
  • ‚ÄúJustBetas‚Äù means that only beta versions can be deleted If no Registry is specified the user will be prompted, unless there is just one anyway.

Note that Tatin interrogates all known Registries for their delete policies just once. It then saves the results on a variable for better performance. This can be a problem is case the policy of a server got changed, most likely your own one. To overcome this specify the -check flag which will result in the servers being asked again rather than trusting former results.

InstallPackages

Syntax

Install the given package(s) and all dependencies into the given folder.

Syntax: 1-2 arguments

Accepts modifier -nobetas

]Tatin. <packageIDs|package-URLs|Zip-file> <install-path>

Details
]Tatin. <packageIDs package-URLs Zip-file> <install-path>

Installs the given package(s) and all dependencies into a given folder.

If the installation folder does not yet exist it will be created; user must confirm.

Requires two arguments:

A) First argument: You may specify a Tatin alias for a package by putting it to the front and separate it with an @. This allows to use different versions of the same package in parallel. Don't use it for other purposes.

If neither a Registry nor a ZIP is specified but just a package ID (partly/fully) then all defined Registries with priority>0 will be scanned; the first hit wins.

(For dependencies a scan will be performed in any case)

Case does not matter: MyGrp-MyPkg, mygrp-mypkg, MYGRP-MYPKG are all considered equal.

A package might be:

  • A full package ID.

    A full package ID has three components: {group}-{name}-{major.minor.patch}.

  • You may also specify an incomplete package ID in terms of no patch number, or neither minor nor patch number, or no version information at all, and leave it to Tatin to establish the latest version itself.
  • You may also omit the group. This will fail in case the same package name is used in two or more different groups but will succeed otherwise.
  • Either a full path or an http[s] URL in front of the package ID.

B) Second argument The optional second argument must be one of:

  • Path to a folder into which the packages are going to be installed
  • ‚Äú[MyUCMDs]‚Äù (case insensitive) with or without specifying a name; the name will be derived from the package ID if none was specified
  • A Cider alias specifying a project

If more than one user command package is specified after [MyUCMDs] an error is thrown.

If no second argument is specified Tatin tries to find an open Cider project.

If there is just one open, Tatin acts on it, otherwise the user is questioned.

It then inspects the ‚Äúdependencies‚Äù & ‚Äúdependencies_dev‚Äù properties.

If that defines just one folder Tatin acts on it.

If there are multiple folders defined the user is questioned which one to install into.

-nobetas: By default beta versions are included. Specify -nobetas to suppress them.

ListPackages

Syntax

List all packages in the Registry or install folder passed as argument

Syntax: up to 1 argument

Accepts modifiers -date, -group=, -noaggr, -os=, -project_url, -since=, -tags=, -uc

]Tatin.ListPackages <URL|[Alias|<path/to/registry>|<install-folder>]> -uc -group= -tags= -os= -date -project_url -since={YYYYMMDD|YYYY-MM-DD} -noaggr

Details
]Tatin.ListPackages <URL [Alias <path/to/registry> <install-folder>]> -uc -group= -tags= -os= -date -project_url -since={YYYYMMDD YYYY-MM-DD} -noaggr

Lists all packages in the Registry or install folder specified.

  • If no argument was specified then the principal Tatin Registry will be assumed (https://tatin.dev)
  • If ‚Äú?‚Äù is specified the user will be prompted for the Registry if there are multiple
  • If ‚Äú[*]‚Äù is specified then ALL registries are questioned.

It does not matter whether you specify / or \ in a path, or whether it has or has not a trailing separator: Tatin is taking care of that.

In case an install folder was specified (rather than a Registry) flags are ignored and just two columns are returned: package name and a Boolean indicating principal packages with a 1.

By default all packages are listed. You can influence the output in several ways: -group={foo} List only packages with the given group name.

-tags=foo,goo List only packages carrying the tags ‚Äúfoo‚Äù & ‚Äúgoo‚Äù.

-uc List only packages that are user commands -os=mac List only packages for the specified operating system(s). Must be a comma-seperated list with ‚Äúwin‚Äù, ‚Äúmac‚Äù, ‚Äúlin‚Äù being valid values.

-since= Must be a date (YYYYMMDD or YYYY-MM-DD) when specified.

List only packages published on that date or later.

Implies -date and ignores -noaggr.

You can also influence the data returned with the following flags: -date Add the publishing date to the output. -noaggr is set to 1 then.

-project_url Add the URL saved in the package config file to the result.

-noaggr By default the output is aggregated. -noaggr prevents that.

ListVersions

Syntax

List all versions of the given package of a given Registry or all Registries with a priority>0

Syntax: 1 argument

Accepts modifier -date

]Tatin.ListVersions <[Registry-URL|[Registry-Alias][<group>]-<name>-[<version>] -date

Details
]Tatin.ListVersions <[Registry-URL [Registry-Alias][<group>]-<name>-[<version>] -date

Lists all versions of the given package.

You may specify a package in several ways:

* ]Tatin.ListVersions [Registry-URL Registry-alias]{group}-{package}
* ]Tatin.ListVersions [Registry-URL Registry-alias]{package}
  • ]Tatin.ListVersions {package}
  • ]Tatin.ListVersions {package}-{major}
  • ]Tatin.ListVersions {package}-{major}.{minor} In all these cases a list of packages is returned, possibly empty.

If just a package is specified, a matrix with two columns is returned: [;1] The Registry [;2] Full package ID

Lacking a group does not make a difference if the package exists only in one group anyway.

If it exists in more than one group then all of them are listed.

If version precedence cannot be established from the version numbers alone (often a problem with beta versions) then the publishing date is taken into account.

-date : adds the publishing date to the report.

Maintenance

Syntax

Checks whether there are maintenance files available and asks the user about it

Syntax: up to 1 argument

Accepts modifiers -dry, -show

]Tatin.Maintenance path -dry -show

Details

]Tatin.Maintenance path -dry -show Checks whether there are any files in the folder Maintenance/. These are expected to be .aplf files (functions). Such files will be listed, and the user can select which one(s) to execute. They will be fed with the argument provided, which is expected to be a folder.

That folder and all its sub folders will be scanned for packages, and every package will be potentially changed by the maintenance file(s) selected by the user.

Usually this means changing the package config file.

This mechanism can be used to, say, rename properties, or inject new ones. The purpose is to ensure that all packages are up to date.

-dry When specified all actions will be listed, including a list of all packages found, but no package will actually be changed at all.

-show Shows the leading comments of all maintenance files that would be executed.

If this is specified no other action is taken.

Note that this is NOT about packages that are managed by a Tatin server. The server has its own mechanism for updating packages.

PublishPackage

Syntax

Publish a package to a particular Tatin Registry.

Syntax: up to 2 arguments

Accepts modifier -dependencies=

]Tatin.PublishPackage <package-folder|ZIP-file> [<Registry-URL|Registry-Alias]> -dependencies=

Details
]Tatin.PublishPackage <package-folder ZIP-file> [<Registry-URL Registry-Alias]> -dependencies=

Publishes a package to a particular Tatin Registry.

Such a package can be one of:

  • ZIP file, typically created by calling ]Tatin.BuildPackage
  • Folder that contains everything that defines a package; in this case the required ZIP is created by ‚ÄúPublishPackage‚Äù itself.

Requires up to two arguments:

  • Path to ZIP file or package folder
  • URL or alias of a Registry or a ‚Äú?‚Äù (you may or may not embrace the ‚Äú?‚Äù with []) In case this is not specified the principal Registry is assumed.

The name of the resulting package is extracted from the ZIP file which therefore must conform to the Tatin rules.

Tatin checks the ‚Äúdelete‚Äù policy of the Registry. If the package cannot be deleted then the user must confirm that she really wants to publish to that particular Registry.

-dependencies=: Use this to specify a subfolder of the project that hosts the packages the package about to be published depends on. Usually there is no need to specify this; refer to the documentation (‚ÄúPublishing Packages‚Äù) for details.

ReInstallDependencies

Syntax

ReInstall all packages installed in a folder from scratch.

Syntax: up to 2 arguments

Accepts modifiers -dry, -force, -nobetas, -update

]Tatin.ReInstallDependencies [install-folder] [Registry-URL|Registry-alias] -force -dry -nobeta -update

Details
]Tatin.ReInstallDependencies [install-folder] [Registry-URL Registry-alias] -force -dry -nobeta -update

ReInstalls all packages (principals as well as dependencies) from scratch.

Takes a folder as argument. If specified the folder must host a file apl-dependencies.txt.

If this is not an absolute path then it might be a sub folder of an open Cider project.

Then Tatin works out the correct one:

  • If there is just one project open it is taken
  • If there are multiple Cider projects open the user is questioned
  • If Cider is not available or no projects are open the current directory is checked For a relative path the user is always asked for confirmation.

If it was not specified at all then all open Cider projects are presented for selection, except if there is only one open anyway. The the Tatin dependency is taken. If there is more than one then they are presented for selection by the user.

All installed packages are removed from the folder (except ZIP files) before a new build list is compiled and used to install all packages from scratch.

Notes:

  • This does not install a later version (but check on -update).

UsageData

Syntax

]Tatin.UsageData [Registry-alias|Registry-URL] -download -all -folder=

Details
]Tatin.UsageData [Registry-alias Registry-URL] -download -all -folder=

Use this to deal with usage data of a Tatin Registry.

If no argument is specified then the user will be prompted for the Registry, except when there is just one anyway.

If no option is specified all available files will be listed.

If only a URL (or Registry alias) is specified a list with all downloadable files is printed.

-download Use this flag to indicate that you wish to download one or more files.

A list with all files available for download will be presented to you together with the additional options ‚Äúall‚Äù and ‚Äúquit‚Äù. You may then select none, one, several or all files for download or cancel the operation.

If you do it must be an empty folder. Ignored when -download is not specified.

The ZIP files will be deleted afterwards. Ignored when -download is not specified.

-all This circumvents the selection dialog; mainly useful for test cases.
